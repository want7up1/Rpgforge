# Prompt / Context 架构重构设计

> 目标读者：接手本项目的工程师（含 Claude）。本文是 token 成本与剧本遵循度的**架构级**重构蓝图，配合 `OPTIMIZATION_PLAN.md`（短周期驾驶舱）使用。
>
> 状态：设计稿（2026-05-29）。阶段 0 已落地（Round 18/19），阶段 1+ 待排期。

---

## 0. 一句话结论

**"更省 token"与"更遵循剧本"在正确架构下是同一个解。** 当前两者都差，根因同一个：把所有信息每回合揉成一坨巨型 user JSON 全量重发。拆开它——按信息的变化速度分层、按 agent 职责按需注入、把机械约束交给代码——两个目标一起达成。

---

## 1. 需求与现状诊断

### 1.1 真实需求（按优先级）

1. **严格遵循剧本强约束**：`hard_rules.must_follow / must_not`、`canon_terms`(专名一致)、角色边界(`public_limit`)、`core_mechanics`、当前幕目标。——用户反复强调"这是最关键的"。
2. **不提前揭露未来剧情**：当前幕不泄露 `next_act` / 未来主线 / `forbidden_reveals`。
3. **降低 token 成本**，尤其 input 端。

### 1.2 现状实测（数据库 79 回合真实 trace）

| Agent | 模型 | 每回合 input tokens | 频率 |
|---|---|---|---|
| story_director | Flash | ~27.5k | 每回合 |
| gm_runtime | Pro | ~26k(流式无 usage，按字符估) | 每回合 |
| state_extractor | Flash | ~31.4k → 投影后降(Round 19) | 每回合 |
| drift_validator | Flash | ~25.3k | 1/3 回合 |
| context_compressor | Flash | ~34.9k → 投影后降(Round 19) | 1/4 回合 |

- **input : output ≈ 30 : 1**，成本几乎全在输入端。
- 旧 GM prompt：SYSTEM 仅 3063 字符通用规则；USER 高达 **60869 字符**。
- USER 构成：`current_state_v2` **41%（24937 字符）**、`runtime_story` 32%、`memory_summaries` 11.4%、`recent_turns` 10.7%。**剧本约束（hard_rules+story_core）合计 <4%，被彻底淹没。**

### 1.3 病根

把三类**变化速度完全不同**的信息平等地塞进一个 payload、每回合 × 每个 agent 重发：

| 层 | 内容 | 变化频率 |
|---|---|---|
| **宪法层（静态）** | GM 角色定义、hard_rules、canon_terms、角色基础人设、core_mechanics、worldview | **整局不变** |
| **幕级简报（半静态）** | 当前幕目标/锚点/允许揭露/禁止揭露 | 每几回合(过幕)变一次 |
| **回合动态** | 玩家输入、状态变化、最近剧情、本回合决策 | 每回合变 |

宪法层和幕简报本该只发一次或极少次，现在每回合重发 → 既费 token，又让约束被每回合变的噪声淹没（模型分不清谁是铁律）。

## 2. 设计原则

1. **按变化速度分层**：稳定的放 prompt 最前（可缓存、地位最高），动态的放后面。
2. **按需注入**：每个 agent 只拿它职责真正用得上的 context（Round 19 已对状态运算 agent 验证）。
3. **约束分级**：机械可判定的（canon/在场/未来揭露）交给代码确定性保证；叙事性的（碾压/五环节）交给 LLM，放宪法层。
4. **缓存友好**：稳定前缀必须**字节级一致**，才能命中 DeepSeek 自动 prefix cache。
5. **增量迁移**：每阶段独立上线、独立验证、可回退；不推倒重来。

## 3. DeepSeek 官方 Context Caching：本架构的核心杠杆

后端确认为 **DeepSeek 官方**，其 Context Caching 特性直接决定支柱 1 的可行性：

- **自动、无需配置**：基于磁盘 KV 缓存，按请求的**最长公共前缀**自动命中。无需传 `cache_control`。
- **命中粒度 64 tokens**：前缀以 64 token 为单位匹配。
- **命中计费大幅降低**：缓存命中的 input token 价格约为未命中的 **1/10**（以官方最新定价为准）。
- **可观测**：响应 usage 返回 `prompt_cache_hit_tokens` / `prompt_cache_miss_tokens`。
- **有效期**：数小时至数天，不活跃自动清除。

**推论（架构约束）**：
- 凡是稳定内容（宪法层 + 幕简报），必须**放在 prompt 最前面**且**逐回合字节一致**（不能掺入时间戳、随机顺序的 dict、每回合变的计数等），否则前缀错位、缓存全 miss。
- 这要求约束注入做到**确定性序列化**（固定 key 顺序、固定文案）。
- ⚠️ **流式调用默认不返回 usage**（见 `OPTIMIZATION_PLAN` 1.1，GM 流式 tokens 现为 None）。要监控 cache 命中，需在流式请求加 `stream_options={"include_usage": true}`——阶段 2 顺带修复，同时补全 GM telemetry。

## 4. 目标架构：四根支柱

### 支柱 1 — 三层 prompt，稳定前缀缓存化
每个 LLM 调用的 prompt 物理排成：`[宪法层] + [幕级简报] + [回合动态]`，稳定层置顶。

- **宪法层**：GM 角色定义 + 剧本叙事性硬约束（must_follow/must_not/canon_terms/角色边界/机制规则）。整局不变 → 命中 cache。
- **幕级简报**：当前幕目标/锚点/允许/禁止揭露。过幕才变 → 同一幕内多回合命中 cache。
- **回合动态**：玩家输入 + 场景状态投影 + 最近剧情 + Director 决策。每回合变 → 必然 miss（本应如此）。

**收益**：省 token（稳定层缓存命中降 ~90%）+ 遵循更好（约束在最顶部、地位稳定、不被淹没）。

### 支柱 2 — 状态用"场景投影"，不发全量
`state_v2` 占 user 41%（实测构成：skills 6.4k / relationship_tracks 3.9k / story_progress 3.8k / quest_log 2.6k / protagonist_sheet 2.1k / progression 1.7k / abilities 1.6k / npc_registry 1.4k / ...）。GM 写当前这场戏，只需要**当前场景相关状态**：在场角色、当前位置、活跃线索、主角关键摘要。

重构：GM 拿到 `state` 的**场景投影**（在场实体 + 活跃线索 + 主角摘要 ≈ 3-5k），而非 25k 全量。完整状态仍在库、由 StateExtractor 维护，但不喂 GM。

**收益**：单点砍掉每回合最大 token 黑洞；GM 注意力聚焦"此刻这场戏"，剧情更聚焦、更少跑题。

### 支柱 3 — 机械约束代码化，叙事约束进宪法
- **机械可判定**（canon_terms 一致、角色在场/位置、不提前揭露未来幕实体、状态数值）：用代码在生成后**确定性校验/纠正**。⚠️ **整串匹配，绝不做滑动窗口子串黑名单**（Round 16 教训）。
- **叙事性**（碾压/进化/五环节描写等）：必须 LLM 执行，放宪法层。

**收益**：prompt 只留"必须靠模型理解"的约束（更省）；机械约束代码兜底（更可靠，模型不会"忘"）。

### 支柱 4 — 剧本设定按相关性注入（RAG 化）
`story_material_library` 已是检索式注入（`retrieve_story_materials`）——推广到 `core_characters`（8 个）、地点、机制：按"本回合是否相关"注入，而非全量。本回合只涉及 2 个角色，就只注入这 2 个的设定。

**收益**：进一步压缩 + 降噪；与支柱 2 的"在场投影"天然契合。

## 5. 重构后 token 画像（估算）

| | 当前 | 重构后(支柱2/3/4，无缓存) | 再叠加缓存(支柱1) |
|---|---|---|---|
| GM input | ~26k 全量 | 动态层 ~10-12k | 稳定前缀命中 → **有效 ~9k** |
| 常驻 3 agent/回合 | ~85k | ~35-45k | **~20-30k** |

- 即使后端无缓存，光支柱 2/3/4（场景投影 + 约束外移 + 检索注入）就能砍 **50%+** input。
- DeepSeek 有缓存，支柱 1 再砍稳定层的 ~90% → 合计可达 **60-75%** input 降幅。

---

## 6. 为什么"遵循剧本"会实质变好（非省钱副作用）

1. **Lost in the middle**：LLM 对长 context **中部**信息利用率显著低于首尾。约束现埋在 6 万字 user 中部 → 天然被忽略。短 context + 约束在 system 顶部 → 命中率大幅提升。
2. **约束地位稳定**：宪法层逐回合字节一致，模型形成"不可违反"的稳定先验。
3. **代码兜底机械约束**：canon/在场/未来揭露不再靠模型自觉。
4. **注意力聚焦**：GM 不被 25k 历史状态分心，专注当前场景 + 宪法约束。

## 7. 分阶段路线图

> 每阶段独立上线、独立验证、可回退。预估收益为每回合 input 降幅的量级估计，需以真实 trace 复核。

### 阶段 0 — 基线（已完成）
- Round 18：剧本强约束提进 GM system prompt（宪法层雏形）。
- Round 19：StateExtractor/Compressor 用精简 runtime_story 投影（支柱 2/4 局部验证，实测 runtime_story 17.5k→6k，省 66%）。
- **作用**：验证"按需注入"方向有效且不破坏功能，为后续阶段铺路。

### 阶段 1 — GM 场景投影 state（支柱 2）｜ 收益★★★ 风险★★
- **目标**：GM 不再吃全量 `state_v2`（25k），改吃**场景投影**。
- **改动**：
  - `state_v2.py` 或新模块加 `project_state_for_scene(state_v2)`：返回在场角色（`active_scene` + `npc_registry` 中在场者）、当前位置、活跃 `open_threads`、`conditions`、`protagonist_sheet` 摘要；剔除非在场 NPC 全量、已完成 quest、历史 progression 明细。
  - `prompt_builder.build_runtime_messages`：GM 的 `current_state_v2` 改用投影。
  - **DriftValidator 仍用较全状态**（判状态冲突需要），或单独投影。
- **验收**：
  - GM user 中 state 部分字符数下降 ≥ 50%。
  - 回归测试：投影保留在场实体/活跃线索/主角摘要；不丢当前场景必需状态。
  - 真实游玩 5-10 回合：剧情未出现"忘记在场角色/当前位置"的状态断裂。
- **风险/回退**：投影过激可能让 GM 丢失需要的状态 → 保守起步（宁可多留），按 trace 观察后收紧。回退 = 改回全量。

### 阶段 2 — 固化宪法层 + 接通 DeepSeek caching（支柱 1）｜ 收益★★★ 风险★★
- **目标**：把 GM（及其他 Pro/Flash 调用）的稳定内容固化成**字节一致的前缀**，命中 DeepSeek prefix cache。
- **改动**：
  - 重排 GM messages：`system = 宪法层(整局稳定)`；user 开头放 `幕级简报(过幕才变)`，再放回合动态。确保宪法层 + 幕简报在 prompt 最前、序列化确定性（固定 key 顺序、固定文案、无时间戳/随机序）。
  - 流式调用加 `stream_options={"include_usage": true}`，拿到 `prompt_cache_hit_tokens`；落入 `agent_traces`（补全 GM 流式 telemetry）。
  - admin dashboard 加 cache 命中率指标。
- **验收**：
  - 同一局连续回合，trace 显示 `prompt_cache_hit_tokens` > 0 且占稳定层比例高（目标命中率 > 80% 的稳定前缀）。
  - 过幕回合命中率短暂下降、之后回升（验证幕简报边界正确）。
- **风险/回退**：前缀若不小心掺入每回合变内容 → 命中率骤降，dashboard 可发现。回退 = 不影响功能，仅失去缓存红利。

### 阶段 3 — 机械约束代码化（支柱 3）｜ 收益★★（遵循↑）风险★★
- **目标**：canon_terms 一致性、角色在场一致、未来幕实体未提前出现，用代码在 GM 输出后确定性校验。
- **改动**：
  - 新增 `canon_checker`：检测 narrative 是否出现 canon_terms 的**错写变体**（基于人工精确指定的别名表，**整串匹配**），或漏用。命中 → 标记 telemetry / 触发轻量重写。
  - 在场/位置一致性校验：narrative 提到的角色是否与 `active_scene` 矛盾。
  - **严禁**从未来幕文本自动生成子串黑名单（Round 16 已否决）。
- **验收**：
  - 注入已知违例（专名错写）的伪输出，校验器能命中。
  - 真实游玩：canon 不一致 / 角色穿越 类问题下降（用 turn_judge 的 canon_fidelity 维度量化前后）。
- **风险/回退**：误报 → 保守阈值 + 只标记不强制重写起步。

### 阶段 4 — 剧本设定 RAG 化（支柱 4）｜ 收益★★ 风险★★
- **目标**：`core_characters` / 地点 / 机制按本回合相关性注入，而非全量。
- **改动**：
  - 扩展 `retrieve_story_materials` 的思路到角色/地点：基于玩家输入 + 当前场景 + 近期剧情检索相关实体，`always_on` 主角恒在。
  - GM 的 `core_characters` 改为"在场 + 检索命中"子集 + 主角。
- **验收**：GM user 中 core_characters 部分字符下降；真实游玩未出现"该出场的角色设定缺失"。
- **风险/回退**：漏召回关键角色 → `always_on` 兜底 + 在场角色强制注入。

### 阶段 5（可选，激进）— agent 职责重划 / Director-GM 评估｜ 收益★ 风险★★★
- **目标**：评估 Director 与 GM 是否可在共享缓存前缀下合并/简化，减少一次 LLM 调用。
- **前置**：必须先有阶段 1-3 + 充足 trace + judge 数据；否则盲目合并损害质量。
- **默认结论**：暂不做（见 `OPTIMIZATION_PLAN` §4 已有"不引入框架/不做大重构"的克制原则）。仅在数据证明 Director 收益低时再议。

## 8. 横切关注点

### 8.1 可观测性（先行）
- 阶段 2 接通 `include_usage` 后，`agent_traces` 能记录 cache hit/miss。
- admin dashboard 增加：每回合平均 input token、cache 命中率、各 agent token 趋势。
- **没有度量就不要优化**：每阶段前后用同一批回合对比 token 与 judge 评分。

### 8.2 遵循度的量化
- 用既有 `turn_judge`（opt-in）的 `canon_fidelity` / `safety` / `state_consistency` 维度，对重构前后同剧本回合打分，量化"遵循是否变好"。
- 不靠主观感觉；每个声称"遵循更好"的阶段都要有评分对比。

### 8.3 回归测试策略
- 每个投影/裁剪函数都是纯函数 → 必有单测（锁定"保留什么、砍什么"）。
- 每阶段改 prompt 结构后，跑全套 + 真实游玩 N 回合。

## 9. 风险总览与原则

| 风险 | 缓解 |
|---|---|
| 投影/裁剪过激，agent 丢失必需信息 | 保守起步（宁多留）、纯函数单测、真实游玩验证后再收紧、可回退 |
| 缓存前缀被污染，命中率骤降 | dashboard 监控命中率；序列化确定性测试 |
| 机械校验误报 | 整串匹配（不滑动窗口）、保守阈值、只标记不强制重写起步 |
| 大重构引入回归 | 增量分阶段、每步独立上线、py 纯函数测试 + 容器 pytest |

**贯穿原则**：吸取 Round 16 教训——**不做字符串子串黑名单做语义判断**；机械约束只用人工精确指定的整串。

## 11. 敲定的优化方案（2026-05-29 头脑风暴收敛）

经一轮收敛式问答，与产品负责人敲定以下决策，作为本文后续执行的**总纲**。

### 11.1 六个决策

| 维度 | 敲定 |
|---|---|
| 产品定位 | **框架内自由**：涌现叙事 + 约束框架。玩家自由行动，但剧本关键节拍/锚点须按序发生、幕目标达成才过幕。**不做 scene/choice 分支节点图，不让 AI 选 transition**（见 11.4）。 |
| 遵循范围 | **四类全要**：风格强约束、剧情走向、设定一致性、状态不出错。 |
| 北极星 | **遵守剧本质量 > token 成本 > 响应速度**。冲突时后两者让位。 |
| 验证器形态 | **轻量校验，不改 AI 输出**：GM 仍输出 narrative，程序生成后读文本校验。 |
| 违规处理 | **只标记不重写**（观测层先行）：违规写 telemetry，不自动干预生成。 |
| 执行顺序 | **遵守与省 token 并行**：两条线不冲突，一起推进。 |

### 11.2 关键认知：四类约束按"可机械判定度"分手段

用户四类全要，但**代码验证器对不同类约束的能力天差地别**，必须分工，不能指望验证器包治：

| 约束类 | 可机械判定 | 兜底手段 |
|---|---|---|
| 设定一致性（canon 专名 / 角色越界 public_limit） | **高** | 验证器**整串**校验 → 标记（绝不滑动窗口，Round 16 教训） |
| 状态不出错（在场/位置/数值/物品） | **高** | StateApplier 按规则算数值（已有）+ DriftValidator 查状态冲突（已有）+ 验证器补在场/位置一致 → 标记 |
| 剧情走向（跑偏/提前揭露/跳幕） | **中** | 锚点驱动过幕（已有代码）+ DriftValidator LLM 语义判偏离（已有）+ 提前揭露检测 → 标记 |
| **风格强约束（战斗碾压 / [剧情规则] / [剧情规则] / 处刑至死）** | **低/不可** | **代码救不了**——无法用程序判断"描写是否覆盖五环节"。只能靠 system prompt 强化（Round 18 已做）+ turn_judge 评分观测 + 必要时调 prompt |

> ⚠️ 用户最痛的"风格强约束没见到"，**验证器对它基本无能为力**。这类的解法是 Round 18（提进 system prompt）+ judge 观测遵守度 + 迭代 prompt，不是代码校验。

### 11.3 验证器 = 观测层（只标记，度量先行）

- 每回合 GM 输出后，代码做轻量校验，把违规写进 telemetry（`turn_jobs` / `agent_traces` / 新列），dashboard 可见违规类型与频率。
- **不干预生成**。先积累"到底违反了什么、多频繁、哪类最严重"的真实数据。
- **未来升级路径（数据驱动，非现在做）**：观测层跑一段时间、数据清楚后，再决定是否对"严重且高频"的机械类违规升级为"分级轻量重写"——届时是数据驱动，不是盲目重写。

### 11.4 明确不做（固化决策，避免反复提议）

- **不把剧本改成 scene/choice/transition 节点状态机**。RPGForge 是涌现叙事（玩家自由输入），不是分支叙事（视觉小说/对话树）。压成节点图要么扼杀自由度（降级成 AI 视觉小说），要么自由输入下节点爆炸无法穷举。剧本保持"幕/锚点/目标"框架——这个粒度适配涌现叙事。
- **不让 AI 输出 `selected_transition` 选剧情节点**。现状"锚点完成 → 代码驱动过幕"比"AI 选 transition"更稳，后者把跳幕权交还 AI、更易幻觉。
- **不做验证器"机械裁剪文本"**。连贯 narrative 切一段会断裂（Round 16 教训）；验证器只标记，未来若干预也是"轻量重写指引"而非裁剪。

### 11.5 两条并行线 + 执行序列（落地为 Round）

**线 A — 省 token（与遵守一致，靠精简而非牺牲）**：支柱 2 GM 场景投影 state（砍 41% 黑洞）+ 支柱 1 宪法层固化接通 DeepSeek prefix cache。

**线 B — 遵守观测**：支柱 3 验证器观测层（canon 整串 + 未授权实体/在场 + 提前揭露检测，只标记）+ turn_judge 评风格类遵守度。

落地序列：

- **Round 20（线 B 起步）✅ 已落地**：验证器观测层 v1（`output_observer.py`）——generation_parameters 达标 + forbidden_reveals 整串命中 + canon 使用度 + 角色提及；**只写 telemetry**（`TurnJob.turn_runtime_inputs.output_observation`）；不干预生成。真实数据已暴露：GM 字数普遍仅达硬下限 70%、canon 14 专名每回合仅用 2-3 个。在场一致性留 v1.1（present_npcs 数据常空）。dashboard 展示 → Round 20b。
- **Round 21（线 A）**：GM 场景投影 state（支柱 2，最大 token 黑洞）。
- **Round 22（线 A）**：宪法层固化为字节稳定前缀 + 接通 DeepSeek prefix cache + GM 流式加 `stream_options.include_usage` 补 token/cache 观测。
- **持续**：turn_judge 评 canon_fidelity / safety / 状态一致性，观测风格类与整体遵守度趋势。

### 11.6 一句话总纲

> **框架由程序与状态机执行并持续观测，AI 负责表达。违规先可观测、再数据驱动治理——不盲目重写，不改产品形态。**
